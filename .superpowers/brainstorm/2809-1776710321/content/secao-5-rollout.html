Seção 5 — Rollout, Sequenciamento e Riscos

Como entregar o v2 em fases, manter o Analytics atual funcionando em paralelo, e mitigar risco.

Ordem de entrega — 5 fases
Fase Dura Entrega Visível pra user?
0 — Preparação silenciosa 2 dias Schema changes (novos campos), indices, triggers, backfill histórico ❌ Não (invisível)
1 — Fundação de dados 5 dias 12 RPCs estendidas (filtros universais) + correção attribution bug + eventos ganho em activities + business hours ❌ Não (APIs internas)
2 — Novas RPCs 6 dias 13 RPCs novas + IA edge function + testes de performance ❌ Não (APIs internas)
3 — Frontend 10-12 dias 5 dashboards persona + barra de filtros + Explorar (pivot + IA) + drill-down transversal ⚠️ Feature flag: só admin vê
4 — Beta com você + ajustes 3-4 dias Você usa por 1 semana, coleta feedback, ajusta widgets que ficaram estranhos, valida números ✅ Sim, só você
5 — Liberação geral + cleanup 2-3 dias Ativar pra gestores → operadores. Depois de 2 sem estáveis, desligar Analytics antigo. Remover RPCs legadas em 1 mês. ✅ Sim, todos

Analytics antigo continua funcionando durante todas as fases. Só é desligado depois que o novo está provado em produção.

Estratégia de não-quebrar Analytics atual
  • RPCs estendidas mantêm assinatura retrocompatível — novos parâmetros com DEFAULT NULL, código antigo continua chamando sem quebrar
  • RPCs corrigidas (attribution) ganham nova versão _v2 — a antiga permanece, frontend novo usa a nova, frontend antigo usa a antiga até migrar
  • Feature flag na rota — /analytics continua apontando pro antigo; /analytics/v2 pro novo. Ao final, redirect do antigo pro novo
  • Dashboards antigos mantidos como pages até desligamento final (user pode comparar novo vs antigo)
  • Testes E2E cobrindo os 2 em paralelo durante a transição
Convivência com modelo legado (proposals vs viagens)

Modelo novo (viagens/trip_items) ainda está em adoção; modelo antigo (proposals/proposal_items) ainda existe pra cards antigos.

  • RPCs de prontidão e versões (analytics_trip_readiness, analytics_proposal_versions) tentam viagens primeiro
  • Fallback pra proposal_trip_plans se card não tem viagem ativa
  • Nenhum card fica sem dado; mas analytics sinaliza visualmente cards "modelo legado" pra o gestor saber que a informação vem de fonte menos rica
  • Quando 100% dos cards migrarem pro novo modelo (projeto paralelo do Travel Planner), removemos o fallback
Testes e qualidade
  • Unit/SQL tests — cada RPC nova ganha caso em business-rules.test.sql (a suite já existe e roda antes de promote)
  • E2E Playwright — fluxos críticos: login → dashboard persona → aplicar filtro → drill-down → Explorar IA → export CSV
  • Performance tests — toda RPC nova precisa retornar em < 2s p95 com 50K cards (teste contra staging com seed realista)
  • Canary numérico — na Fase 4 (beta), você valida se os números do v2 batem com o Analytics antigo em métricas compartilhadas (ex: receita fechada no mês). Se divergir, investigar antes de liberar
  • Regression suite — Analytics antigo continua com testes E2E até o desligamento (impede quebra acidental)
Riscos e mitigações
Risco Mitigação
Queries do Explorar pesadas → app lentoIndices específicos + cache 5 min + limite 1000 linhas + período máx 24m
IA interpreta errado → gestor lê número errado"Entendi assim" sempre visível + pivot editável sempre + confidence score
Correção attribution muda números históricos → gestor estranhaNota explicativa na tela: "Rankings agora consideram o dono da fase correta (antes usava dono atual — incorreto)"
Backfill gera load pesado no banco em produçãoExecutar em batches de 1K, off-hours, com pausas. Um campo por vez
Feature flag esquecido → operador vê sem quererFlag controlada por role no profiles.is_admin + checklist de liberação
Custo IA descontroladoRate limit: 30 queries/user/hora inicialmente. Cache de interpretações repetidas.
V3 NPS — próximo projeto (outline, não entra no v2)

NPS é projeto próprio. Escopo estimado ~5-7 dias:

  • Tabelas nps_requests + nps_responses
  • Edge function de envio automático N dias após retorno (usa data_viagem_fim)
  • Página pública /nps/{token} pra cliente responder (sem login)
  • Cadência de follow-up se não responder em X dias
  • Widgets já reservados nos dashboards v2 (placeholders viram dados reais)
  • Analytics de NPS por Planner, destino, período, correlação com retorno
Resumo final do escopo v2

Entrega:

  • 5 dashboards dedicados por persona (Dono, Comercial, Vendas, Pós, SDR)
  • Barra de filtros universais (7 filtros + Modo/Lente)
  • Aba Explorar híbrida (pivot + IA texto transparente)
  • Drill-down transversal com timeline por card
  • Fundação backend (12 RPCs estendidas, 5 corrigidas, 13 novas, business hours, 2 edge functions IA)
  • Feature flag + rollout seguro mantendo Analytics antigo durante transição

Não entrega (fica pra depois):

  • NPS completo (tabela + envio + captura + analytics) — V3
  • Áudio no Explorar + refinar multi-turn — V2.5
  • IA proativa / drill em dashboards via IA — V3
  • Email/telefone SLA — requer instrumentação nova, projeto futuro

Estimativa: ~30 dias úteis, 5 fases, rollout com feature flag, manutenção de compatibilidade total com Analytics antigo durante transição.
