Subagentes agora rodam em Opus 4.8
✅ testado ao vivo

O conserto-mãe. Era o que mais afetava qualidade — e ninguém via.

🔧 Como o Claude Code escolhe o modelo de um auxiliar

Existe uma ordem de prioridade fixa. Quem está mais no topo vence:

1
Configuração CLAUDE_CODE_SUBAGENT_MODEL
← MANDA EM TUDO
2
O modelo passado em cada chamada (model: "opus")
sobrescrito pelo nº 1
3
O modelo definido no próprio auxiliar
—
4
O modelo do agente principal
—

É por isso que a regra antiga do projeto ("sempre force Opus na mão") nunca funcionou: ela mexia no nº 2, mas o nº 1 estava em Haiku e vencia.

📝 O que eu mudei (antes → depois)

🔴 ANTES · só no global, em Haiku
// ~/.claude/settings.json
"env": {
  "CLAUDE_CODE_SUBAGENT_MODEL":
      "claude-haiku-4-5..."
}
// → TODO auxiliar = Haiku
🟢 DEPOIS · no projeto, em Opus
// WelcomeCRM/.claude/settings.json
"env": {
  "CLAUDE_CODE_SUBAGENT_MODEL":
      "claude-opus-4-8"
}
// → TODO auxiliar = Opus 4.8

A configuração do projeto vence a do global — então isso vale só no WelcomeCRM, sem afetar seus outros projetos.

🔬 A prova (registro real do teste)

Disparei um auxiliar de teste e li o modelo gravado no registro dele:

transcript do subagente de teste · campo "model" $ grep '"model"' subagents/agent-a732...jsonl
"model":"claude-opus-4-8" ← nasceu Opus ✅ (antes vinha "claude-haiku-4-5")
Como funciona no dia a dia: você não muda nada no seu jeito de pedir. Toda vez que eu disparo um auxiliar pra investigar ou revisar, ele agora nasce Opus 4.8 sozinho. Custa um pouco mais lento/caro — que é o "Opus em tudo" que você queria.

A memória do projeto agora chega nos auxiliares
✅ testado · 4 cenários

O conserto de maior valor estrutural. Antes, os auxiliares investigavam "cegos".

🔧 O problema, visual

O "buscador de contexto" (a ferramenta que orienta cada agente) lia o mapa do código, mas não lia a pasta de memória (as 102 anotações de erros e decisões). E os auxiliares de investigação nem recebem o manual. Resultado: repetiam erros que o projeto já tinha aprendido.

⚙️ Como funciona agora (o fluxo, passo a passo)

1 · Tarefa
"criar uma migration no banco"
→
2 · Buscador
lê as palavras da tarefa
→
3 · Casa nas 102 notas
palavra do nome do arquivo pesa 3× a da descrição
→
4 · Entrega as regras
devolve as 3-4 notas certas (nome + resumo)

Como o auxiliar pode chamar o buscador, ele recebe essas regras sozinho — não preciso mais colar nada na mão. É o pulo do gato: a memória passou a ser "servida" por uma ferramenta que todos alcançam.

🔬 A prova (o que cada tarefa traz de volta — testado)

Se a tarefa é sobre……o buscador entrega a regra certa
migration / subir pro banco→feedback_migration_cleanup.md
listar usuários do workspace→feedback_multi_tenant_org_members.md
campo no pipeline errado→feedback_metadata_isolation.md
funil de analytics weddings→feedback_analytics_weddings_apenas_active.md
Por que isso importa: são justamente as regras que mais viram bug. Agora qualquer auxiliar que eu disparar para mexer nessas áreas já chega sabendo — não repete o erro.
3
O índice de memória voltou a carregar inteiro
✅ verificado

Parte das suas regras estava ficando invisível no começo de cada sessão.

🔧 Como funciona o carregamento

O Claude Code carrega só as primeiras 200 linhas do índice de memória quando começa. O que passa disso, não carrega. O índice estava com 229 linhas — então as últimas 29 (regras de fluxo, manutenção, ferramentas) sumiam.

linha 1 … começo do índice
linhas 2–200 · regras carregadas ✅
— — — corte em 200 linhas (a "janela" que carrega) — — —
linhas 201–229 · 🔴 ANTES: não carregavam (Workflow/Estilo, Manutenção, Ferramentas)

📝 O que eu mudei

Compactei a seção "Trabalho Ativo" (de bloco grande por projeto → 1 linha de status + ponteiro), sem perder nenhum ponteiro do índice. Resultado:

🔴 ANTES
229 linhas
→ últimas 29 invisíveis
  no início da sessão
🟢 DEPOIS
193 linhas
→ tudo cabe na janela de 200
→ índice inteiro carrega ✅
  (99 ponteiros preservados)

Um "comece aqui" para qualquer agente novo
✅ criado

Antes, o conhecimento estava espalhado em 4 lugares sem um mapa.

⚙️ O que o START-HERE faz

Um arquivo único (docs/START-HERE.md) que diz, em uma tela: os 4 lugares de conhecimento e quando usar cada um, como achar contexto rápido, as regras que mais viram bug, e o modelo dos agentes. O manual principal agora aponta pra ele logo no topo.

Detalhe honesto: o sistema travou num avisinho de qualidade que não era da minha tarefa — um atalho (goToBloco) marcado como "exportado" sem precisar, na sua tela de planejamento de casamento. Removi só essa marca (uma palavra), sem mudar o funcionamento, e deixei a mudancinha solta pra ir junto com a sua feature.
Para ligar tudo: o conserto nº 1 (Opus) já está valendo. Os outros (memória no buscador + índice compacto) ligam quando você fecha e reabre o Claude Code uma vez.
