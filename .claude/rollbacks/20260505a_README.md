# Undo da feature "Concierge no card detail" (20260505a)

Documenta como desfazer cada etapa, em qualquer estado da implantação.

## O que foi alterado

1. **Banco** — quatro migrations cumulativas:
   - `20260505a_auto_create_atendimento_for_concierge_assignee.sql` —
     cria o trigger e a função (dispara só em INSERT na primeira versão).
   - `20260505b_auto_create_atendimento_on_assignee_change.sql` —
     amplia o evento do trigger para INSERT OR UPDATE OF responsavel_id,
     cobrindo o caso "planner reatribui tarefa existente para concierge".
   - `20260505c_trigger_reads_tipo_from_metadata.sql` —
     a função passa a ler `tipo_concierge` e `categoria_concierge` do
     `tarefas.metadata` (preenchido pelo SmartTaskModal). Defaults seguem
     iguais (operacional/outro) para casos sem metadata.
   - `20260505d_drop_planner_request_source.sql` —
     remove o valor `'planner_request'` da CHECK constraint de
     `atendimentos_concierge.source`. Toda criação humana fica como
     `'manual'`. Backfill defensivo (UPDATE → 'manual') é no-op em prod
     porque a distribuição já era cadencia=11/manual=10/cliente=4/planner_request=0.
     Rollback dedicado em `20260505d_rollback.sql`.
   - `20260505e_trigger_skips_cadence_metadata.sql` —
     o trigger passa a pular quando a tarefa veio de cadência/automação
     (detecta `metadata.cadence_instance_id`, `cadence_step_id`,
     `automation_rule_id`, ou `metadata.origin in (cadence, automation,
     event_trigger)`). Conserta race condition que estava marcando
     atendimentos como `'manual'` quando deveriam ser `'cadencia'`.
     Inclui backfill defensivo: linhas com source='manual' cuja tarefa
     tem sinais de cadência/automação foram movidas para `'cadencia'`
     (2 linhas em prod).
     Rollback dedicado em `20260505e_rollback.sql` (volta a função pra
     versão sem a guarda — só use se a nova lógica regredir).
   As primeiras três mexem no trigger `tarefas_auto_create_concierge_atendimento`
   e na função `trg_tarefas_auto_create_concierge_atendimento`. A quarta
   mexe só na constraint da coluna `source`. A quinta mexe na função
   do trigger novamente.

2. **Frontend** (branch `feat/concierge-card-implant-realtime`):
   - `src/components/card/CardTasks.tsx` — selo Concierge na lista, clique abre modal de atendimento, canal realtime, badge de outcome (Aceito/Feito/Recusado/Cancelado)
   - `src/components/card/SectionConciergeOverview.tsx` — canal realtime adicional
   - `src/components/card/SmartTaskModal.tsx` — seletor obrigatório de tipo (Oferta/Reserva/Suporte/Operacional) quando o responsável é do time Concierge; passa o tipo escolhido em `metadata.tipo_concierge`

## Estados possíveis e como reverter

### Estado A — antes de subir para produção (situação atual)

Nada está em produção. A feature vive só no branch local + staging.

**Undo:** voltar para `main` e descartar o branch.

```bash
git checkout main
git branch -D feat/concierge-card-implant-realtime
```

O ajuste no banco de **staging** continua aplicado (não atrapalha — staging não tem o módulo Concierge ativo, e o trigger tem guarda defensiva).

---

### Estado B — migration já em produção, frontend ainda não

Desfaz só o trigger. Nenhuma tela mudou, então não tem nada a reverter na UI.

```bash
bash .claude/hooks/promote-to-prod.sh .claude/rollbacks/20260505a_rollback.sql
```

Tarefas que o trigger já criou continuam visíveis na aba Concierge — são dados reais
(o operador pode já ter mexido). Não são apagadas.

---

### Estado C — migration em produção E frontend já em main

1. Reverter o trigger no banco:
   ```bash
   bash .claude/hooks/promote-to-prod.sh .claude/rollbacks/20260505a_rollback.sql
   ```

2. Reverter o frontend no Git. Achar o commit de merge da feature e reverter:
   ```bash
   git checkout main
   git pull
   git log --oneline | grep -i "concierge.*card.*implant\|concierge-card-implant-realtime" | head -5
   # pegar o hash do commit de merge ou do squash, depois:
   git revert <hash> -m 1
   git push
   ```
   O Vercel vai redeploy a versão sem o selo + modal + realtime do CardTasks.
   `SectionConciergeOverview` volta a ler com staleTime de 30s (sem realtime).

3. As linhas de `atendimentos_concierge` que o trigger criou continuam no banco
   e seguem aparecendo em `/concierge`. Se quiser auditar quais foram criadas
   nesse intervalo:
   ```sql
   SELECT id, tarefa_id, card_id, created_at
   FROM atendimentos_concierge
   WHERE source = 'manual'
     AND created_at >= '<data-do-deploy-em-prod>'
     AND created_at <= '<data-do-rollback>'
   ORDER BY created_at DESC;
   ```
   Não dá pra distinguir 100% dessas linhas das que humanos criaram manualmente
   no mesmo intervalo (mesmo `source='manual'`). Por isso o rollback não apaga
   nada automaticamente.

## Notas

- A guarda `to_regclass('public.atendimentos_concierge')` no trigger garante
  que ambientes sem a tabela (staging defasado, branches de teste) não quebram
  inserts em `tarefas`. Mesmo deixando o trigger ativo lá, é seguro.
- O rollback SQL é idempotente: pode ser rodado mais de uma vez sem erro.
